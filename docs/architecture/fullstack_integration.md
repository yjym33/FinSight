# 풀스택 통합 흐름 분석서 (시니어 아키텍트 관점)

## 1. 전체 시스템 통합 아키텍처 개요
본 시스템은 지능형 주식 분석 플랫폼으로서, 프론트엔드의 세련된 UI가 백엔드의 실시간 금융 데이터 및 AI 처리 능력을 어떻게 효과적으로 소비하는지에 중점을 둡니다.

## 2. 통합 데이터 파이프라인 (End-to-End Flow)

### 일반적인 종목 상세 조회 요청 케이스
1. **[UI Step]**: 사용자가 삼성전자(005930)를 클릭합니다.
2. **[Client Step]**: Next.js 페이지 이동(`stocks/[code]`) -> React Query가 REST API(`/api/stocks/005930/price`)를 호출합니다.
3. **[Server Step]**: 백엔드 컨트롤러가 요청 수신 -> `StocksService`가 가동됩니다.
    - `KisService`가 KIS 시스템에서 시세를 가져옵니다.
    - `AnalysisService`가 OpenAI에 분석을 요청합니다.
4. **[Integration Step]**: 백엔드는 API 응답과 별개로 WebSocket 연결을 통해 수밀리초마다 업데이트되는 시계열 체결 데이터를 클라이언트로 스트리밍합니다.
5. **[Render Step]**: 프론트엔드는 AI 분석 결과가 도착하면 `AIStockAnalysisCard`를 애니메이션과 함께 노출합니다.

## 3. 핵심 통합 기술: AI 그라운딩(Grounding) 전략
- **분석 문제**: AI가 특정 종목의 상세 사업 내용을 정확히 인지하지 못하는 '할루시네이션' 현상.
- **해결 방식**: 
    - 백엔드 DB에서 해당 종목의 **정확한 업종(Sector)** 정보를 가져와 프롬프트의 컨텍스트(Hint)로 직접 주입.
    - "이 종목의 업종은 '반도체'이다. 이를 기반으로 사업 개요를 작성하라"는 명확한 지침 하달.
- **결과**: 외부 공식 프로필 API 없이도 매우 정확한 기업 요약 정보를 사용자에게 제공합니다.

## 4. 인프라 안정성 계층
- **중앙 집중형 호출 제어 (Distributed Throttling)**: 유저가 수만 명으로 늘어날 때 외부 KIS API 계정이 차단되지 않도록 Redis 기반 중앙 제어 장치를 운영합니다.
- **환경 변수 안전 장치**: 서버 부팅 시점에 DB URL, KIS Key, OpenAI API Key의 유효성을 전수 조사하여 잘못된 환경에서의 기동을 원천 차단합니다.
- **인증 무결성**: JWT 기반의 전역 인터셉터를 통해 프론트엔드와 백엔드의 보안 흐름을 동기화합니다.

## 5. 아키텍처적 한계 및 미래 제안
- **네트워크 지연**: AI 분석 시간의 물리적 지연(약 2~4초)을 사용자에게 즐겁게 보여주기 위한 'AI 타이핑 애니메이션' 등 감성적인 로딩 처리 추가 제안.
- **상태 동기화**: REST 요청 시점과 WebSocket 수신 시점 간의 시차를 보정하기 위해 클라이언트 측 시계열 데이터 버퍼링 레이어 도입 검토 필요.
- **DB 정합성**: 복잡한 비즈니스 로직(커뮤니티, 알림 등)에 대해 백엔드 트랜잭션 수립과 수신 클라이언트 측의 Optimistic Update(낙관적 업데이트) 적용 필요.
